# BitingLip Cluster Manager: Progress & Architecture Report

## Current Development Progress of Cluster Manager

The **Cluster Manager** in the BitingLip platform has made significant progress and is now marked “Production Ready.” All core features outlined in its roadmap have been implemented and tested. Key implemented capabilities include:

- **Real-Time Monitoring:** The manager continuously tracks system resources (CPU, memory, disk) on a 30-second interval. This provides up-to-date metrics on each node’s load and health.
- **Database Persistence:** A PostgreSQL database is integrated (with connection pooling) to persist cluster state. The cluster manager automatically initializes the schema (tables for nodes, resources, events, GPUs) on first startup, ensuring all node registrations and metrics are stored reliably.
- **Node Lifecycle Management:** Nodes register automatically and send periodic heartbeats. The cluster manager registers itself as the **master node** on startup and cleans up stale node entries if a node fails to report in. This means the foundation for multi-node support (tracking multiple worker nodes) is in place.
- **GPU Detection & Multi-GPU Support:** On each node, GPUs are detected automatically and their availability is recorded. The system supports **multi-GPU setups** – the design has provisions for utilizing multiple GPUs for tasks. Configuration options like `GPU_DEVICES` allow specifying which GPU IDs are available to the cluster manager’s workers. This was tested with comprehensive test suites to validate GPU detection and node registration.
- **Distributed Task Processing:** The cluster manager leverages **Celery** (with Redis as the message broker) for distributing AI inference tasks to GPU workers. In the current design, Celery worker processes (running on the GPUs) execute tasks, and the cluster manager orchestrates these workers. The use of Celery provides a robust framework for task queuing, scheduling, and result collection (with a result backend configured).
- **Cluster Health & Analytics:** It provides cluster-wide health metrics and basic analytics on resource usage. This includes aggregate CPU/memory utilization and possibly simple load balancing decisions based on these metrics (e.g. which node has capacity). Basic fault tolerance is in place (e.g. detecting unresponsive nodes). The platform’s **Gateway** and **Task Manager** services can query the cluster manager (directly or via the database) to get this health information for routing or scheduling decisions.
- **Robustness & Testing:** The codebase has been cleaned up and extensively tested. Recent commits added **comprehensive test suites** for GPU detection, database operations, node registration, and the monitoring loop. A “cleanup” refactor removed obsolete components (e.g. an older standalone worker script) and reorganized code for clarity. The final structure is modular and maintainable, with core logic under `app/`, utilities under `utils/`, and documentation under `docs/`. The development team confirms the cluster manager is now a **“production-ready service with a clean, maintainable codebase.”**

Overall, the current Cluster Manager fulfills its initial goals: it acts as the **orchestration hub** for the GPU cluster, managing worker nodes and coordinating tasks. It runs as a microservice within the BitingLip platform and interfaces with other managers (Gateway, Task, Model) through database updates and possibly API endpoints. In the platform’s architecture, it occupies the orchestration layer alongside the Task and Model Managers, and it bridges to the execution layer where the actual GPU **worker processes** run. The cluster manager’s status in the platform is “✅ Production Ready,” indicating that it has been tested and is considered stable for use.

## Current Architecture and Responsibilities

**Architecture Overview:** BitingLip uses a **modular microservices architecture** for AI model serving. The key managers include:

- **Gateway Manager** (API gateway, port 8001) – front-end for requests and authentication.
- **Model Manager** (model registry and storage, port 8002) – handles model files and metadata.
- **Task Manager** (task queue and scheduling, port 8004) – queues inference requests, manages task priorities.
- **Cluster Manager** (cluster orchestration, port 8083) – manages GPU nodes/workers and load balancing.
- **GPU Workers** (execution layer) – the processes that run the ML models on GPUs.

These services communicate primarily via REST APIs and shared infrastructure (Redis, database). The microservice design is intended to maximize flexibility – for example, one can deploy a minimal set of services on a worker node (just what’s needed to run tasks) while the central services run elsewhere.

**Cluster Manager’s Role:** The cluster manager serves as the **control plane** for the GPU cluster. Its responsibilities currently span both **node management** and **worker management**:

- At the cluster level, it maintains the authoritative registry of all nodes in the cluster (which nodes are active, their resources, etc.) in the PostgreSQL database. It monitors the health of each node through heartbeats and updates the database or internal state accordingly. This data can be used by the Gateway/Task managers to make scheduling decisions or display cluster status.
- At the node level, the cluster manager also directly handles the local machine’s workers. In the current implementation, the “master” cluster manager process runs on one node and launches/coordinates worker processes (Celery workers) on that same node to utilize its GPUs. The cluster manager’s codebase includes task definitions and a worker management component to start or interface with these GPU worker processes. Essentially, the cluster manager on the master node doubles as a **node manager** for its own machine – it detects local GPUs, starts local Celery workers (which run the actual model inference tasks), and monitors their resource usage.
- **Celery Integration:** The cluster manager uses Celery for distributed task execution. All Celery workers (on one or multiple nodes) subscribe to a Redis-based queue (managed by the Task Manager) and execute AI inference tasks. The cluster manager ensures that worker processes are running on available GPUs and can consume tasks. Celery’s built-in scheduling handles delivering tasks to any available worker, while the cluster manager oversees the availability of those workers. This setup provides basic load balancing: if multiple worker processes (across one or several nodes) are listening on the queue, Celery will distribute tasks among them. However, any **cluster-level load balancing logic** beyond Celery’s round-robin is currently limited – e.g., the Task Manager might not yet direct tasks to specific nodes based on load (it just hands tasks to Redis). The cluster manager does update resource metrics (CPU/GPU utilization), which could later be used to influence scheduling decisions.
- **Persistence & Coordination:** The cluster manager is stateful. It persists node and resource info in the database and possibly exposes a service API (e.g., endpoints for retrieving cluster status or updating node info). Other managers, like the Gateway Manager, can query cluster status – for example, the Gateway might call the cluster manager (or read its DB entries) to show an overview of active nodes and their loads. The Model Manager may also interact so that when a new model is deployed, cluster manager knows each node’s model loading status (though model placement is primarily the Model Manager’s domain).
- **Current Deployment Mode:** At present, running the cluster manager in a single-node scenario involves starting the cluster manager service and its workers on the same machine. The documentation’s quick start shows the cluster manager being launched and automatically handling its local worker processes. For multi-node use, it appears that one would designate one instance as the “master” cluster manager (running the orchestration + DB update loop) and run _some subset of the cluster-manager code on the other nodes as well_ – likely just the worker portion. (Originally, the repository had a `cluster/worker` submodule for running a worker on each node, but this was removed and restructured in the latest refactor.) The current architecture doesn’t yet have a separate lightweight agent; instead, additional nodes might need to run the cluster manager script in a mode where it registers the node and runs Celery workers (but without becoming a second master). This setup is somewhat implicit – the “master” knows about other nodes if they register in the DB, but **the mechanism for other nodes to register now is likely manual or via running a portion of the code on those nodes.** This is an area poised for improvement, as described next.

## Need for Node Manager & Improved Worker Management

While the cluster manager is functionally complete, its **scope is currently broad** – it is managing the entire cluster _and_ directly managing workers on the local node. This all-in-one approach works for a single-node deployment or a small cluster, but it can be improved by further modularizing into a hierarchy of managers:

- **Cluster Manager vs Node Manager:** Splitting the responsibilities between a **Cluster Manager** (global view) and a **Node Manager** (per-node agent) is a logical next step. In the current design, the cluster manager handles “node management” itself – meaning it contains code for GPU detection, starting local workers, updating local metrics, etc., in addition to maintaining cluster-wide state. Introducing a dedicated **Node Manager microservice on each worker node** would offload those local duties. Each Node Manager would be a lightweight service responsible for a single node’s operations: detecting that node’s GPUs, launching/monitoring worker processes on that node, and sending heartbeat and resource usage reports to the central cluster manager. Meanwhile, the **Cluster Manager** would focus only on the global cluster state – aggregating info from node managers, maintaining the cluster-wide database, making high-level decisions (like which node should get a new workload), and serving as a single entry point to query cluster status. This separation follows a proven design in distributed systems (similar to a master/agent pattern found in Kubernetes or Spark, where a central master schedules work and per-node agents actually execute the work).

  - **Benefits of a Node Manager:** By having a minimal agent on each node, the system becomes more **scalable and flexible**. You could deploy the BitingLip platform to new machines by simply running the Node Manager (plus perhaps the Model runtime) on them – they would auto-register with the Cluster Manager’s API. The cluster manager would no longer need direct database access from every node; instead, node managers could communicate over an HTTP or gRPC API. This reduces coupling – the database writes go through the cluster manager service, allowing validation and easier schema updates. It also means the cluster manager process itself can run just on a primary node or control server, and worker nodes run a lean process with only the essentials. In effect, the **“minimal software package”** for nodes that the team envisions would likely include the Node Manager and whatever is needed to run models, but not the full orchestrator logic.

  - **Node Manager Responsibilities:** The node manager would handle operations like starting/stopping **GPU worker processes**, monitoring GPU memory/utilization on its machine, and possibly handling local events (e.g. if a GPU goes offline or a process crashes). It would implement the heartbeat to the cluster manager: for instance, every 30 seconds (or a configurable interval) it could POST its node’s stats to a cluster manager endpoint (rather than the cluster manager polling everything). It could also receive commands from the cluster manager, such as “start a new worker for model X” or “download model Y” (though model distribution might be handled by the Model Manager separately). Offloading these tasks means the cluster manager doesn’t need to SSH or directly control remote nodes; it delegates to the Node Manager via a clear API.

- **Worker Manager or Direct Worker Control:** The idea of a separate “worker-manager” (one level below the node manager) has been proposed. This would imply an additional microservice or module that specifically manages an individual GPU or a pair of GPUs. In practice, we should carefully consider if this extra layer is necessary. Each **GPU worker** process (for example, a Celery worker process bound to a specific GPU) can be managed directly by the Node Manager without an intermediate microservice. The Node Manager can simply spawn one worker process per GPU, or in cases where a single task needs multiple GPUs, spawn a process that grabs multiple GPUs. Introducing a dedicated “worker manager” per GPU would mean running yet another service (likely just to supervise one process), which might be over-engineering the solution.

  - **GPU Pairing and Multi-GPU Tasks:** One challenge is handling tasks that require multiple GPUs (for example, large model inference that spans 2 GPUs). To address this, we have two approaches:

    1. **Single Worker, Multiple GPUs:** Launch a worker process that is configured to utilize a specific set of GPUs (e.g., two GPUs in one process). This could be controlled by environment variables like `CUDA_VISIBLE_DEVICES` or via a configuration that the Node Manager passes to the worker. The Node Manager would need to ensure that those GPUs are free, then start a worker that claims both. No separate “worker manager” needed – the Node Manager orchestrates this grouping logic.
    2. **Multiple Workers, Coordinate at Task Level:** Alternatively, one could have two separate worker processes (each on one GPU) cooperate on a task, but that is more complex and typically would require a higher-level scheduler (and perhaps a distributed training framework). For inference serving, it’s usually simpler to use one process for the multi-GPU task.

  Given these options, implementing **GPU pairing logic in the Node Manager** is preferable. The Node Manager already has a global view of its local resources, so it can allocate one or more GPUs to a new worker process as needed. We likely do **not** need a standalone “worker-manager” service per GPU; instead, the Node Manager can directly manage the lifecycle of workers. Each worker is essentially a process (or container) running the model inference code (e.g., a Celery worker or similar) – this process can be started, stopped, or restarted by the Node Manager. If we were to introduce a microservice for each GPU, it would essentially be doing the same job (start/stop a process) but with extra overhead (each microservice would need its own API, etc.). Thus, keeping this layer simple is best: treat the **worker** as a lightweight process, not a full service. The Node Manager can maintain a small internal map of which GPU(s) each worker process is using, and handle pairing by launching combined-GPU workers when required.

In summary, **I recommend splitting out a Node Manager and _not_ adding an extra worker-manager microservice**, unless a clear need arises. The Node Manager can cover the functionality of managing one or many workers on its node. This design will allow the **Cluster Manager** to become truly focused on multi-node coordination. It will unify multiple nodes under one control point, as intended, without being burdened by per-node details.

## Future Improvements and Roadmap Suggestions

With the core features in place, the project should now focus on **architectural refinement and foresight** to support growth. Below are suggested improvements and future features, centered mostly on the management layer (as the interfaces like CLI/GUI are outside our scope):

- **Implement the Node Manager Service:** As discussed, creating a dedicated Node Manager component is a top priority. This will involve:

  - Designing a **Node Manager API** (endpoints or RPC) that the Cluster Manager will use to communicate with nodes. For example, endpoints like `/heartbeat`, `/metrics`, `/spawn_worker`, `/stop_worker` on the Node Manager. Initially, even just the heartbeat/metrics endpoint would allow the cluster manager to stop actively monitoring and instead accept data from nodes.
  - Packaging the Node Manager as a minimal deployable service. This might mean extracting the relevant parts of cluster-manager code (GPU detection, worker launch, etc.) into a new repository or module. The Node Manager would likely run a very lightweight web server (FastAPI or even a small Flask app, etc.) or possibly a persistent Celery beat process for heartbeats – but a REST API would be more flexible.
  - Security considerations: ensure that the communication between Cluster Manager and Node Managers is authenticated (since it’s a distributed system, maybe use tokens or mutual trust, especially if deployed across machines).

- **Refine Multi-Node Coordination:** With Node Managers in place, the Cluster Manager can truly manage multiple nodes. We should implement:

  - **Auto-Registration of Nodes:** When a Node Manager starts, it should register itself with the Cluster Manager (supplying details like node name, available GPUs, etc.). This could be as simple as the Node Manager calling a `/register` endpoint on the Cluster Manager. The cluster DB’s `cluster_nodes` table would get a new entry and the cluster manager marks it active.
  - **Heartbeat and Failure Detection:** Rather than the Cluster Manager polling each node’s status via direct DB or internal threads, switch to a model where each Node Manager sends a heartbeat message. The cluster manager can mark a node unhealthy if heartbeats stop. This is more scalable as the number of nodes grows.
  - **Distributed Monitoring:** The current 30-second monitoring loop is fine for one node, but for many nodes, it should be distributed. Each Node Manager can collect its own stats (via `psutil` or NVML for GPU stats) and send them upward. The cluster manager aggregates these to compute cluster-wide metrics (e.g., total free GPUs, overall utilization). This also reduces load on the master, as it’s not making system calls for every node – each node handles its own monitoring.

- **Load Balancing & Task Scheduling Enhancements:** At present, Celery is used to distribute tasks in a somewhat generic way. We can improve how tasks get allocated to workers:

  - **Intelligent Task Routing:** Implement logic in the Task Manager (or integrated with Cluster Manager) to route tasks to specific node-queues based on availability. For example, instead of one global Celery queue for all workers, we could maintain one queue per node or use Celery routing keys for each node or GPU. The Cluster Manager (knowing each node’s load and free GPUs) can inform the Task Manager which node should handle the next task. The Task Manager can then enqueue the task with a routing key targeting that node’s workers. This ensures a busy node isn’t overloaded if others are idle, improving utilization and response times.
  - **Resource-Aware Scheduling:** As a future feature, scheduling decisions can consider more metrics – e.g., if a model is already loaded in GPU memory on Node A, prefer scheduling similar requests to Node A (to take advantage of caching). The Cluster Manager and Model Manager together could track which models are loaded where. This avoids redundant model loads and warms caches effectively.
  - **Auto-Scaling Workers:** With Node Managers controlling worker processes, we could add dynamic scaling policies. For instance, if a node’s queue backlog grows, the Node Manager could spawn additional worker processes (if the GPU has memory headroom to perhaps run multiple models or if using multi-threading) or load another model onto a free GPU. Conversely, if a worker is idle for a long time, the Node Manager might spin it down or unload the model to free memory. This is similar to Kubernetes Pod autoscaling but at the process level for GPU tasks. (This might be an advanced feature; initially one worker per GPU is fine, but it’s something to consider as the platform matures.)

- **High Availability & Redundancy:** The current design relies on a single Cluster Manager as the master. In the future, consider options for high availability:

  - A standby cluster manager instance that can take over if the primary fails (this would require some consensus or failover mechanism and a shared database).
  - Alternatively, run the cluster manager as a replicated service behind a load balancer for read endpoints, but elect one leader for write operations to the DB (to avoid conflicts). This ensures the cluster doesn’t have a single point of failure at the control plane.

- **Enhanced Monitoring & Logging:** While basic metrics are collected, the platform could benefit from deeper observability:

  - **Distributed Tracing:** Implement tracing (using something like OpenTelemetry) across the microservices. For example, an inference request trace could start at Gateway, go through Task Manager, then Cluster Manager, to a Node Manager and worker. This would help diagnose performance issues or bottlenecks. It’s mentioned as a goal in the docs (observability with tracing).
  - **Central Logging:** Ensure all Node Managers and the Cluster Manager feed logs to a central location (or at least are easily accessible) for debugging and auditing. This might involve using a logging server or simply structuring logs in the database.
  - **Alerts and Health Checks:** Expand the `/health` checks. Currently, managers have simple health endpoints. We could create a cluster-wide health summary (perhaps the Gateway or Cluster Manager can expose an endpoint that returns overall status, including if any node is down or any worker has failed). Integrating this with an alerting system (email/Slack alerts when a node goes down) would be valuable in production settings.

- **Streamlined Model Distribution:** While the Model Manager handles model storage and the workers likely download models on-demand (e.g., from the HuggingFace Hub or a central storage), coordination with the cluster management could be improved. For example, if a new model is registered, the Cluster Manager might instruct specific Node Managers to pre-download or load that model into GPU memory if those nodes are designated to serve that model. This pre-warming could reduce latency on first requests. A **node-local caching** strategy, possibly guided by the Cluster Manager, would optimize multi-node model serving.

- **Testing & Validation:** Continue expanding the test suite to cover multi-node scenarios. The recent tests cover GPU detection and monitoring on one node. Future tests should simulate multiple Node Managers registering with the Cluster Manager, tasks being sent to different nodes, and failure recovery (e.g., simulate a node crash and ensure the cluster manager correctly marks it and perhaps redistributes its tasks). A **“chaos testing”** approach could be useful: intentionally kill a worker process or drop a node’s connection to see if the system recovers gracefully.

- **Documentation & Developer Experience:** As architecture changes (like adding Node Manager), update the documentation to reflect the new design. Diagrams should illustrate the hierarchy: e.g., Cluster Manager -> Node Manager -> Workers. Clear instructions to deploy a new node will help users adopt the scaling features. Also, providing examples of the API calls between cluster and node managers will be important for developers extending or integrating the platform.

By implementing the above improvements, the BitingLip platform’s cluster component will become more robust and flexible. The shift to a **multi-tier management architecture** (global cluster manager + per-node managers + per-GPU workers) will resolve the current overlap of responsibilities and prepare the system for larger scale deployments. Each piece (Cluster, Node, Worker) will have a clear, single responsibility, aligning with microservice best practices.

In conclusion, the **overall software architecture** is shaping up to be well-structured: an API-driven microservice ecosystem with specialized managers and distributed workers. The Cluster Manager already provides a strong foundation for cluster orchestration, and with the planned Node Manager separation, it will achieve even better modularity and scalability. These enhancements will ensure that as the platform grows (more models, more GPUs, more nodes), the architecture can handle it gracefully without becoming a monolith. The development progress so far is excellent – the platform is nearly feature-complete for an MVP – and the suggested refinements will keep the momentum going by improving maintainability and performance for the future.
